# Road4AI Launch Plan

*Created: 2026-04-21*

## Objective

Launch Road4AI as a recognizable zero-cost AI operator blueprint for technical builders who want local-first, CLI-native workflows instead of subscription-heavy AI products.

Primary conversion:

- Get builders to run the stack, inspect the repo, and follow Road4AI for future drops.

Secondary conversions:

- Capture email or waitlist signups if a landing page is added
- Drive GitHub visits, stars, and shares
- Create a repeatable launch motion for future releases

## Positioning

### Core message

Road4AI helps technical builders stop renting intelligence and start operating their own zero-cost AI stack.

### Angle

- Anti-SaaS-tax
- Pro-operator
- Local-first
- Practical, not aspirational

### What makes this launch work

- It has a clear enemy: recurring subscription dependence
- It has a clear identity: operators over consumers
- It has a clear artifact: a repo, a blueprint, and verification tooling

## Recommended launch phases

## Phase 1: Foundation

Goal:

- Tighten the message before pushing traffic

Checklist:

- Rewrite README headline and opening section around the operator thesis
- Publish homepage/landing page copy
- Keep `.agents/product-marketing-context.md` current
- Prepare 3 demo assets:
  - repo verification example
  - zero-cost architecture visual
  - CLI workflow screenshot or short terminal capture

## Phase 2: Soft Launch

Goal:

- Test narrative resonance with a technical audience

Actions:

- Post one manifesto-style thread on X
- Post one LinkedIn version framed around AI cost control and workflow ownership
- Share to 3-5 relevant communities where technical operators already hang out
- Ask early readers two questions:
  - What part feels most useful?
  - What part still feels too abstract?

Success signals:

- Replies quoting the core phrases back
- Questions about setup and workflow
- Saves, reposts, and repo visits

## Phase 3: Public Launch

Goal:

- Turn Road4AI into a concrete movement instead of a one-off repo share

Launch day assets:

- Updated README
- Homepage or landing page
- Launch thread
- LinkedIn post
- 60-90 second demo clip
- Hacker News post

Launch day sequence:

1. Publish the main launch post on X.
2. Publish the LinkedIn version 30-60 minutes later.
3. Publish demo clip and link back to the repo.
4. Submit to Hacker News with a technical framing, not a hype framing.
5. Reply actively for the next 6-8 hours.

## Phase 4: Post-Launch Compounding

Goal:

- Keep the idea alive through repeated useful releases

Weekly cadence:

- 1 manifesto or opinion post
- 1 technical tutorial
- 1 build-in-public artifact
- 1 proof post showing a script, result, or benchmark

Release triggers:

- New script
- New workflow
- New architecture revision
- New integration
- New repo audit example

## Risks

### Risk: the message feels ideological but not concrete

Response:

- Always pair manifesto language with repo artifacts, scripts, screenshots, or architecture diagrams.

### Risk: "zero-cost" gets read as low quality

Response:

- Frame zero-cost as disciplined engineering, not cheapness.

### Risk: audience is too broad

Response:

- Keep the first wave focused on technical operators, not general AI users.
